# YGGDRASIL（ユグドラシル）

プロダクトの意思決定を、**3つの異なる観点を持つAIの合議**でサポートする仕組みです。Webアプリやオーケストレーション用のコードは不要で、Cursorの既存機能（カスタムサブエージェント＋スキル＋MCP）だけで動きます。

名前は北欧神話の世界樹に由来します。その根元で運命の女神ノルンの三姉妹——**ウルズ（過去）・ヴェルザンディ（現在）・スクルド（未来）**——が運命を紡ぐように、3体のAIがそれぞれの時間軸からプロダクトの決定を検討します。

| 愛称 | 技術ID | 観点 | データソース | 使用モデル |
|---|---|---|---|---|
| **ヴェルザンディ**（現在） | `data-analyst` | いまプロダクトで起きていることを定量データで語る | `knowledge/data/` | `gpt-5.3-codex` |
| **スクルド**（未来） | `customer-advocate` | ユーザーの声＝これから満たすべきものを代弁する | `knowledge/customer/` | `claude-sonnet-5-thinking-high` |
| **ウルズ**（過去） | `internal-context` | すでに紡がれた糸——ビジョン・予算・経緯——を語る | `knowledge/internal/` + `decisions/` | `grok-4.5-fast-xhigh` |

司会（オーケストレーター）はメインのチャットエージェントが務めます。愛称は**ラタトスク**——世界樹を駆けて言葉を運ぶリスです。サブエージェント同士は直接会話できないため、ラタトスクが3体の間で意見を運びます。

## 使い方

Cursorのチャットで論点を投げるだけです。

```
/decide ○○機能を廃止すべきか？
```

司会が次のプロトコルで議論を進行します。

1. **Round 1**: 3体を並列起動。各AIは自分のデータソースだけを調べ、出典付きで意見を返す
2. **Round 2**: 各AIに他の2体の意見を渡し、自分の証拠に基づく反論・補強・立場の更新、そして**転向条件**（何が示されれば立場を変えるか）を求める
3. **統合**: 推奨・各観点の根拠・**解消されなかった不一致**・不足情報・転向条件をまとめた決定メモを `decisions/` に、発言全文の議事録を `decisions/logs/` に出力する

最終判断はあくまで人間が行う前提です。YGGDRASILは「3つの観点の根拠と対立点を短時間で揃える装置」であり、不一致を隠さず出させることに価値を置いています。

## 設計の要点

**「観点」ではなくデータソースを物理的に分離**しています。各AIは担当ソースにない事柄を「判断できない」と答えるよう縛られており、3体の意見が同じ結論に収束するのを防ぎます。モデルも役割適性に加えて**あえて3ベンダーを混在**させ、思考の癖レベルでの多様性を確保しています（変更は `.cursor/agents/*.md` の `model:` を書き換えるだけ）。司会は対立を丸めずに統合する最も難しい役割のため、チャット本体には上位クラスのモデルを推奨します。

**議論は蓄積され、次の議論に参照されます。** ウルズのデータソースには `decisions/` が含まれるため、過去の合議の結論が「過去の経緯」として自動的に次回へ持ち込まれます。決定メモの「人間の最終判断」欄に実際の決定を追記していくと、AI推奨と実判断の乖離も記録され、参照の質が上がります。

## ディレクトリ構成

```
.cursor/
├── agents/            # 3体の観点AIの定義（役割・データソース・モデル・出力形式）
└── skills/decide/     # 合議の進行手順（/decide で起動）
knowledge/             # ★あなたのプロダクトの資料をここに置く（初期状態は空）
├── data/              #   アクセス解析・KPI・DBのエクスポートなど
├── customer/          #   アンケート・問い合わせ・NPSなど
└── internal/          #   ビジョン・予算・過去の経緯など
decisions/             # 決定メモが蓄積される（初期状態は空）
└── logs/              # 議事録（各AIの発言全文）
examples/taskflow/     # 架空プロダクトのサンプルデータと合議の実例
```

## 自分のプロダクトで使う

1. `knowledge/` の3ディレクトリに手持ちの資料（Markdown、CSV等）を置く
2. `/decide` で論点を投げる

ローカルファイルの代わりにMCPで実データへ接続することもできます。`.cursor/mcp.json` にMCPサーバーを設定してください。各エージェント定義には「MCPツールが利用可能な場合はそちらを優先する」旨を記載済みです。

| 観点 | MCP接続の例 |
|---|---|
| ヴェルザンディ | Google Analytics、BigQuery/Postgres（**必ず読み取り専用で**） |
| スクルド | Zendesk / Intercom / アンケートツール |
| ウルズ | Notion / Google Drive / 社内Wiki |

## サンプル（TaskFlow）で試す

架空のタスク管理SaaS「TaskFlow」のダミーデータを同梱しています。「ガントチャート機能を廃止すべきか？」という論点で3観点の意見が実際に対立するよう設計されています。

```bash
cp -r examples/taskflow/knowledge/ knowledge/
```

その後 `/decide ガントチャート機能を廃止すべきか？` を実行してください。このサンプルで実際に行った2回の合議の記録（2対1で割れた例・反論ラウンドで収束した例）が `examples/taskflow/decisions/` にあり、出力イメージの参考になります。

## 既知の制約

- Cursorの仕様上、サブエージェント単位でMCPツールのアクセス制限はできません（プロンプトによる役割の制約のみ）。DB等は接続自体を読み取り専用にしてください
- サブエージェント同士は直接会話できないため、司会がresumeで意見を仲介するラウンド制です（議事録が残る利点でもあります）
- 1回の議論で3体×2ラウンドのエージェント実行が走るため、相応の時間とコストがかかります
